# Profile Surface Audit

## Current Surface Problems

- The profile `README.md` exposes multiple competing entry points under `Start Here`.
- `verifiable-agent-demo` is still presented beside the architecture hub instead of being classified as a walkthrough surface.
- `fdo-kernel-mvk` appears in the layer table but not as a first-class visible core repository in the front-door narrative.
- The current `Core Projects` section mixes the five-layer core with the demo surface.
- Supporting and legacy repositories are not separated clearly enough from the current five-layer stack.

## Public Narrative Risks

- The account front page does not enforce one canonical interpretation point.
- Demo and supporting repositories can compete with the theory hub for attention.
- The execution-integrity layer is visually underrepresented, which weakens the perceived completeness of the five-layer stack.
- Legacy lineage is not explicitly marked as historical context, so older DPP and sovereignty work can still read like parallel current directions.

## Internal Consistency Issues

- `docs/pinned-front-door-final.md` still recommends pinning `verifiable-agent-demo` instead of restoring `fdo-kernel-mvk` to visible core status.
- `docs/public-surface-ops-checklist.md` still reflects the older pinning logic.
- `docs/repository-priority-model.md` still treats the public line as hub plus POP plus demo plus ARO rather than the five core layers.

## Required Fix

- Enforce a single canonical entry: `digital-biosphere-architecture`.
- Rebuild the profile around `core`, `supporting annexes`, `demo and evaluation`, and `legacy lineage`.
- Restore `fdo-kernel-mvk` to explicit front-page core status.
- Move demo and evaluation to a separate section.
- Move legacy repositories into an explicit historical-lineage section.
